## B1: #50 DTO 接受 cron 表达式(方案 A)

**问题**: `ScheduleConfigDto.times` 强制 `HH:MM`,致 `*/5 * * * *` 类标准 cron 永远无法到达 scheduler(已支持)。

**方案 A: 同一字段、per-item 条件校验**

```typescript
// create-task.dto.ts
import { validate, IsString, IsIn, ValidateIf, ValidationArguments } from 'class-validator';
import { validateCronExpression } from 'cron';

const HHMM_REGEX = /^([01]\d|2[0-3]):[0-5]\d$/;

function isValidTimeOrCron(value: unknown): boolean {
  if (typeof value !== 'string') return false;
  if (HHMM_REGEX.test(value)) return true;
  return validateCronExpression(value).valid;
}

export class ScheduleConfigDto {
  @IsString({ each: true })
  @ValidateIf((_o, v) => v !== undefined && v !== null)
  @IsIn(['__placeholder__'], { each: true })  // 占位,真正的 per-item 校验走 custom
  times: string[];

  // ... 其余不变
}
```

实际更简单的写法 — 用自定义 validator:

```typescript
import { registerDecorator, ValidationOptions } from 'class-validator';
import { validateCronExpression } from 'cron';

function IsTimeOrCron(options?: ValidationOptions) {
  return function (object: object, propertyName: string) {
    registerDecorator({
      name: 'isTimeOrCron',
      target: object.constructor,
      propertyName,
      options,
      validator: {
        validate(value: any) {
          if (!Array.isArray(value)) return false;
          return value.every((v) => {
            if (typeof v !== 'string') return false;
            if (HHMM_REGEX.test(v)) return true;
            return validateCronExpression(v).valid;
          });
        },
        defaultMessage: () =>
          'each `times` must be HH:MM (00:00–23:59) or a valid 5-field cron expression',
      },
    });
  };
}

export class ScheduleConfigDto {
  @IsTimeOrCron()
  times: string[];

  @IsIn(ALLOWED_TIMEZONES as unknown as string[])
  timezone: string;
}
```

**理由**:
- 不破坏前端 — `times` 字段名不变
- 复用 scheduler 已有 `timeToCron()` 逻辑(已支持两类输入)
- 错误消息明确,400 即可返
- 不引入新字段、不拆 DTO,改动局部

**验证**:
1. `times: ['*/5 * * * *']` → 201
2. `times: ['08:00', '*/15 * * * *']` 混合 → 201
3. `times: ['25:00']` → 400
4. `times: ['bogus cron']` → 400
5. `times: ['', null, 123]` → 400

## B2: #56 e2e — pipeline notification 失败隔离

**目标**: 真跑一遍 pipeline 步骤,验证 notification 失败时:
- `Run.status === 'SUCCESS'`
- `AutoPublishArticle.status === 'PUBLISHED'`
- notification 失败被 catch,log warn,return 不 throw
- 错误消息被记录到 `Run.errorLogs` 或 `Article.failedStep`(具体字段看 schema)

**测试文件**: `backend/test/pipeline-notification-isolation.e2e-spec.ts`

**策略**:
- 用 `Test.createTestingModule({ imports: [AppModule] })` 真实启动
- Mock `NotificationStep.execute` 抛错(`Test.createTestingModule` 用 `useValue` override 不可行 — 用 `overrideProvider`)

**依赖**: 需要 MySQL + Redis 测试数据。复用 `app.e2e-spec.ts` 的 beforeAll/afterEach cleanup 模式。

**风险**: Pipeline 依赖 AI/WordPress/SMTP,需要 mock 大量 providers。**降级方案**: 如果 e2e 启动过重,改用单元测试扩展 `pipeline.service.spec.ts`(已 4/4 PASS,加 1 case: 把 notification step 注入 throw,断言 PipelineService 不 rethrow)。

## B3: #52 e2e — research P95 性能

**目标**: 真接口触发 `POST /stories/:id/research` 5 次,断言:
- 5 次全成功(HTTP 201)
- P95 耗时 < 60s(原基线 120s,改进后应远低)

**测试文件**: `backend/test/research-perf.e2e-spec.ts`

**策略**:
- 真实数据(story + 关联 article)通过 prisma 注入
- 串行触发 5 次(并行会限速 API)
- `Date.now()` 测每轮耗时
- 计算 P95,assert < 60_000ms

**注意**:
- #52 真接口 A 阶段已 PASS(已返真实 timeline)
- 主要价值是 **性能 SLA 承诺的回归保护**
- 如果 AI API 不可达(网络/限速),test skip

**风险**: 5×60s = 5min,超 Jest 默认 timeout。需 `jest.setTimeout(600_000)`。
